Panduan Lisensi Platform ERM

Sistem lisensi terpusat untuk produk Gemilang Inti Teknologi — komprehensif untuk admin, user aplikasi, dan developer.

Overview Sistem

Platform ERM dilindungi oleh sistem lisensi yang terdiri dari dua aplikasi yang berkomunikasi via HTTPS:

AplikasiPeranStackDatabaseURL Production
Gemilang License Server (penerbit lisensi) Laravel 13 + PostgreSQL license license.gemilangteknologi.com
ERMv3 Aplikasi ERM (client lisensi) Laravel 13 + PostgreSQL ermv3 erm.gemilangteknologi.com

Konsep Inti

  • License Key — kode unik berformat LIC-XXXX-XXXX-XXXX-XXXX yang diterbitkan Gemilang dan diberikan ke client untuk mengaktifkan ERMv3.
  • Fingerprint — sidik device yang dihitung dari hostname + machine UUID + APP_KEY + database identity. Setiap fingerprint unik per server. Lisensi terikat ke fingerprint, jadi tidak bisa dipakai di server lain tanpa aktivasi ulang.
  • PASETO Token — token kriptografis (Ed25519 signed) yang diterbitkan server ke client setiap aktivasi/refresh. Tidak bisa dipalsukan tanpa private key server.
  • Heartbeat — ping berkala dari ERMv3 ke Gemilang untuk verifikasi lisensi masih aktif. Kalau gagal terus-menerus, ERMv3 akan kunci diri sendiri.
  • Feature License Key (FLK) — kunci tambahan opsional untuk fitur premium di dalam ERMv3. Format: FLK-XXXX-XXXX-XXXX-XXXX.

Cryptography

  • Algoritma: PASETO v4 public mode + Ed25519 detached signature
  • Root Key: 1 key — bertugas mensign signing key (chain of trust)
  • Signing Key: 1 active key (rotasi 30 hari) — bertugas mensign token aktivasi
  • Public Key: ada di storage/app/licensing/keys di Gemilang. Dibagikan ke client via API /api/platform/v1/public-key.
  • Tidak ada secret di .env client — license_key + public_key disimpan encrypted di storage/app/licensing/state/ ermv3.

ADMIN Login Gemilang

  1. Buka https://license.gemilangteknologi.com/login
  2. Masukkan kredensial super admin:
    • Email: admin@gemilang.local
    • Password: gemilang2025 (ganti segera setelah login pertama)
  3. Setelah login akan masuk ke Dashboard dengan menu: Home, Master Data (Companies, Apps, Configs, Feature Flags), Licensing (Licenses, Installations), Users.
⚠ Best Practice
Buat user admin baru sesegera mungkin lewat menu Users → New, lalu nonaktifkan akun super admin default agar tidak mudah di-bruteforce.

ADMIN Tambah Company Client

Setiap perusahaan klien yang akan menggunakan ERM diregister sebagai MasterCompany dulu sebelum bisa dibuatkan lisensi.

  1. Master Data → Companies → klik + New Company
  2. Isi:
    • Name — nama PT/CV (misal "PT. ABM Investama")
    • Code — kode singkat unik (misal "abm")
    • Email, Phone, Address — informasi kontak
    • Status — Active / Inactive
  3. Klik Save. Company siap dipilih saat membuat lisensi.

ADMIN Daftarkan Aplikasi (App)

Setiap produk yang akan dilisensikan didaftarkan sebagai MasterApp. Default sudah ada ermv3.

  1. Master Data → Apps → + New App
  2. Isi:
    • Code — kode unik aplikasi (lowercase, tanpa spasi). Code ini yang dikirim client di header request. Misal: ermv3, finsys.
    • Name — nama display.
    • Description — opsional.
  3. Klik Save.

ADMIN Setup Feature

Setiap aplikasi punya daftar fitur. Fitur bisa Free (cuma toggle on/off) atau Licensed (butuh kunci aktivasi tambahan / FLK).

  1. Master Data → Apps → klik aplikasi (misal ermv3)
  2. Scroll ke section Features → + Tambah Fitur
  3. Isi:
    • Feature Key — identifier unik (misal dashboard, export-pdf). Dipakai di Blade @feature('dashboard').
    • Name — nama display
    • Category — pengelompokan (Dashboard, Reports, Admin, dll)
    • Requires License — centang kalau fitur ini premium dan butuh FLK key tersendiri
  4. Klik Save.
💡 Free vs Licensed Feature
  • Free — admin bisa toggle on/off di Master / Per-License. Client otomatis dapat akses.
  • Licensed — generate FLK key dulu, lalu kirim ke client untuk aktivasi via menu Feature License di ERMv3.

ADMIN Terbitkan Lisensi

  1. Licensing → Licenses → + New License
  2. Isi form:
    • Company — pilih company client (yang sudah didaftarkan)
    • Label — opsional, untuk identifikasi internal (misal "ABM Production")
    • Days — durasi berlaku dalam hari (365, 730, dst)
    • Max Installations — berapa device boleh aktivasi pakai kunci ini (default 1)
    • Apps — centang aplikasi mana saja yang termasuk lisensi ini
    • Notes — catatan internal
  3. Klik Save. Modal tampil kunci LIC-XXXX-XXXX-XXXX-XXXX.
  4. Salin kunci dan kirim aman ke client (email + WhatsApp). Kunci hanya muncul sekali — kalau lupa, gunakan fitur "Tampilkan Kunci" atau "Generate Kunci Baru".
⚠ Max Installations
Default 1. Jika client memiliki staging + production, set ke 2. Setiap aktivasi memakan 1 slot. Slot bisa dibebaskan dengan revoke (lihat section Revoke).

ADMIN Lisensi Per-Fitur (FLK)

Untuk fitur yang punya Requires License = true, perlu generate kunci aktivasi terpisah berformat FLK-XXXX-XXXX-XXXX-XXXX.

  1. Master Data → Apps → klik aplikasi → scroll ke Features
  2. Pada baris fitur premium, klik Generate FLK → modal tampil kunci FLK
  3. Salin FLK dan kirim ke client. Client masuk ke ERMv3 menu License → Feature Activation dan masukkan FLK.
  4. Setelah aktivasi, fitur otomatis bisa digunakan.

Toggle Aktif/Nonaktif Per License

  1. Buka detail license di /licenses/{hash}
  2. Section Licensed Apps & Features menampilkan tabel fitur
  3. Klik tombol Suspend / Aktifkan per fitur

ADMIN Revoke / Suspend / Cancel / Renew

Revoke Single Usage

Hapus satu instalasi (slot kembali kosong, client itu harus aktivasi ulang):

  1. Buka /licenses/{hash} → section Installation Slots
  2. Klik tombol Revoke pada card instalasi yang ingin di-revoke
  3. Konfirmasi → row license_usages di-hard-delete (slot bebas)

Revoke Semua Usage

Bersihkan semua slot sekaligus (semua client harus aktivasi ulang):

  1. Section Installation Slots → Revoke Semua Usage
  2. Konfirmasi

Suspend License

Lisensi tidak dihapus tapi client tidak bisa aktivasi/heartbeat:

  1. License Info → klik Suspend
  2. Untuk reaktivasi, klik Reinstate di tempat yang sama

Cancel License

Permanent — client kehilangan akses, tidak bisa di-reinstate (harus buat baru):

  1. License Info → klik Cancel → konfirmasi

Renew License

Perpanjang masa aktif:

  1. License Info → input Days di kolom Renew (default 365)
  2. Klik Renew — expires_at + N hari

Hapus Lisensi (Soft Delete)

  1. Di list /licenses klik tombol Hapus
  2. Halaman konfirmasi tampil daftar usage aktif yang akan ikut di-revoke
  3. Centang konfirmasi + klik Ya, Hapus Lisensi
  4. Lisensi soft-deleted, semua usage hard-delete (data audit tetap di logs)

ADMIN Regenerate Kunci (Lupa atau Bocor)

Tampilkan Kunci Lama (Recovery)

  1. Buka /licenses/{hash} → section Kunci Lisensi Lupa?
  2. Opsi 1 — klik Tampilkan Kunci → modal tampil kunci lengkap

Generate Kunci Baru

Kalau kunci dicurigai bocor, generate kunci baru. Otomatis revoke semua usage aktif sehingga client harus aktivasi ulang.

  1. Section Kunci Lisensi Lupa → Opsi 2 → klik Generate Kunci Baru
  2. Halaman konfirmasi → input alasan (opsional)
  3. Klik Ya, Generate Kunci Baru
  4. Modal tampil kunci baru — salin dan kirim ke client
🚨 Konsekuensi Generate Kunci Baru
Semua client yang sedang aktif akan kehilangan akses langsung. Pastikan client siap menerima kunci baru sebelum klik tombol ini.

ADMIN Tuning Heartbeat & Policy

Default heartbeat policy bisa di-override per-license.

  1. Buka /licenses/{hash} → section Heartbeat Policy Override
  2. Set:
    • Failure Tolerance — berapa kali heartbeat gagal sebelum client tampil banner warning (default 3)
    • Warning Days — setelah berapa hari banner warning, ERMv3 lockout dengan modal full-screen (default 3)
  3. Klik Save Policy. Client akan dapat policy baru di heartbeat berikutnya.

Global Defaults (Master Configs)

Master Data → Configs → key:

  • licensing.heartbeat_interval — interval heartbeat (detik, default 3600)
  • licensing.heartbeat_retry_limit — default 3
  • licensing.warning_days — default 3
  • licensing.grace_period_days — default 7
  • licensing.timeout — HTTP timeout client → server (default 30s)
  • licensing.cache_ttl — cache token client (default 3600s)
  • licensing.clock_skew_seconds — toleransi perbedaan waktu (default 60s)
  • licensing.issuer — issuer string PASETO (default gemilang-inti-teknologi)

CLIENT Aktivasi ERM Pertama Kali

  1. Pastikan ERMv3 sudah ter-install di server (lihat panduan deploy untuk DevOps)
  2. Buka URL aplikasi (misal https://erm.client.com) di browser
  3. Otomatis redirect ke halaman Aktivasi Lisensi
  4. Server URL Gemilang sudah otomatis terisi dari konfigurasi (tidak perlu diisi)
  5. Masukkan kunci lisensi yang diterima dari Gemilang (format LIC-XXXX-XXXX-XXXX-XXXX)
  6. Klik Aktivasi Lisensi
  7. Tunggu 2-5 detik. ERMv3 akan:
    • Fetch public key dari Gemilang
    • Fetch client config (interval heartbeat, dll)
    • Kirim kunci + fingerprint device ke server untuk verifikasi
    • Terima PASETO token + bundle
    • Simpan semua di storage/app/licensing/state/ (encrypted)
    • Trigger heartbeat pertama
    • Redirect ke halaman License Status
  8. Status menjadi VALID. Klik ← Back untuk akses aplikasi normal.

CLIENT Aktivasi Fitur Tambahan (FLK)

Untuk fitur premium yang dijual terpisah:

  1. Login ke ERMv3 sebagai admin
  2. Menu License → Feature Activation
  3. Pilih fitur yang ingin diaktifkan dari daftar
  4. Masukkan kunci FLK yang diterima dari Gemilang
  5. Klik Aktivasi Fitur
  6. Status fitur menjadi "Activated" — bisa langsung dipakai

Untuk deactivate (sementara), klik tombol Deactivate. FLK key tetap valid dan bisa diaktifkan ulang nanti.

CLIENT Cek Status Lisensi

  1. Akses https://erm.client.com/license/status kapan saja
  2. Atau dari menu Settings → License (jika tersedia di app)

Yang Ditampilkan

  • Status: VALID / INVALID
  • License ID, Status, Max Usages
  • Expires At, Issued At, Not Before
  • Issuer (harus gemilang-inti-teknologi)
  • License Expires At — kapan kunci expired
  • Force Online After — sampai kapan token offline berlaku
  • Usage Fingerprint device

CLIENT Troubleshoot

GejalaKemungkinan PenyebabSolusi
Status: INVALID Token expired atau corrupted Buka /license/install, masukkan kunci ulang
"License key is invalid" Kunci salah ketik, atau kunci sudah di-regenerate di Gemilang Cek email/WA dari admin Gemilang untuk kunci terbaru
"Fingerprint already in use" Ada usage lama dengan fingerprint sama yang belum di-revoke Hubungi admin Gemilang untuk Revoke Usage di /licenses/{hash}
"License usage limit exceeded" Slot installations penuh (max_installations terpakai) Admin Gemilang naikkan max_installations atau revoke usage lama
"Invalid public key format" Public key di storage corrupt atau format tidak match Hapus storage/app/licensing/state/.public_key lalu aktivasi ulang
Banner warning di atas aplikasi Heartbeat sudah gagal ≥3x Cek koneksi internet ke license.gemilangteknologi.com
Modal lockout muncul Heartbeat gagal lebih dari warning_days (3 hari) Pulihkan koneksi, tunggu heartbeat berikutnya, atau aktivasi ulang
"Certificate chain verification failed" Root key di server berubah Aktivasi ulang dari install page (akan fetch bundle baru)

DEV Deployment

Gemilang (License Server) — Pertama Kali

# 1. Clone
cd /www/wwwroot
git clone https://github.com/gtek2022/ermv33.git license.gemilangteknologi.com
cd license.gemilangteknologi.com

# 2. Install
composer install --no-dev --optimize-autoloader

# 3. Setup .env
cp .env.example .env
nano .env  # set DB credentials, APP_URL, dll

# 4. App key + migrations
php artisan key:generate
php artisan migrate
php artisan db:seed --class=DatabaseSeeder

# 5. Generate Ed25519 keys (PENTING!)
php artisan licensing:keys:make-root
php artisan licensing:keys:issue-signing
php artisan licensing:keys:list  # verify

# 6. Cache
php artisan optimize
⚠ Backup keys segera
storage/app/licensing/keys/ berisi private key Ed25519 yang tidak bisa di-recover. Backup folder ini ke cold storage segera setelah generate. Kalau hilang, semua client harus aktivasi ulang dengan kunci baru.

DEV Fingerprint Generator

HardenedFingerprintGenerator menggabungkan multiple sources:

fingerprint = sha256(
  hostname        +  // gethostname()
  machine_id      +  // /etc/machine-id (Linux), wmic UUID (Windows)
  "php-" + version +
  "laravel-" + version +
  "app-" + sha256(APP_KEY).substr(0, 32) +
  "db-" + sha256(host + ":" + database)
)

Setiap component yang berubah akan invalidate fingerprint:

  • Cloning ke server lain → machine_id beda → invalid
  • Ganti APP_KEY → invalid (force re-aktivasi)
  • Pindah database → invalid
  • Upgrade Laravel/PHP → invalid (rare, biasanya minor patch tidak ganti major version)

DEV Heartbeat Setup (Backend & Frontend)

Heartbeat adalah mekanisme verifikasi berkala antara client (ERMv3) dan server (Gemilang). Sistem berlapis dua: backend scheduler (server-side cron) untuk verifikasi tanpa user, dan frontend poller (browser-side JS) untuk responsiveness UI.

1. Backend Scheduler Setup

Heartbeat dijadwalkan via Laravel scheduler. Pastikan cron Laravel aktif di server:

# Edit crontab system
crontab -e

# Tambahkan baris ini (jalan setiap menit, scheduler decide actual interval):
* * * * * cd /www/wwwroot/erm.client.com && php artisan schedule:run >> /dev/null 2>&1

Verifikasi schedule terdaftar:

php artisan schedule:list

# Output yang diharapkan:
# 0 * * * *  php artisan license:heartbeat ............... Next due: in 23 minutes

Konfigurasi Schedule di bootstrap/app.php

Schedule otomatis di-register oleh package laravel-licensing-client. Tapi kita bisa override interval di routes/console.php atau via app schedule:

// routes/console.php atau di App\Console\Kernel::schedule
use Illuminate\Console\Scheduling\Schedule;
use Illuminate\Support\Facades\Schedule as ScheduleFacade;

ScheduleFacade::command('license:heartbeat')
    ->hourly()                  // default — bisa diganti
    ->withoutOverlapping(15)    // skip jika previous run masih jalan
    ->onFailure(function () {
        \Log::warning('License heartbeat scheduled task failed');
    });
💡 Interval Heartbeat
Default 1 jam (3600 detik). Bisa diatur via licensing.heartbeat_interval di Master Configs Gemilang. Setelah diubah server-side, client akan dapat update via client-config endpoint dalam 6 jam berikutnya.

Manual Trigger via CLI

# Jalankan heartbeat sekarang juga (untuk testing/debug)
php artisan license:heartbeat

# Output sukses:
# [License] Heartbeat OK. Status: ok

# Output gagal:

2. Backend Service Architecture

Komponen backend yang saling terkait:

FileTanggung Jawab
App\Services\Licensing\HeartbeatService Core logic: ping(), getStatus(), getStatusData(), refreshPolicy()
App\Services\Licensing\LicenseStateStore Persist failure counter, last_ok timestamp, policy ke encrypted file
App\Console\Commands\LicenseHeartbeatCommand Artisan command yang dijalankan scheduler
App\Http\Controllers\Api\LicenseHeartbeatController HTTP endpoint untuk frontend (status + ping)
App\Http\Middleware\EnforceLicense Cek $status === 'lockout' → flag session, layout render modal

State yang Dikelola

HeartbeatService memakai LicenseStateStore untuk simpan state berikut (semua di storage/app/licensing/state/ encrypted):

FileIsiUpdate saat
.hb_failuresCounter consecutive failures+1 saat fail, reset 0 saat OK
.hb_first_failISO timestamp first fail di streak iniSet saat counter dari 0 → 1, hapus saat OK
.hb_last_okISO timestamp last successful heartbeatSet setiap kali OK
.hb_policyJSON: {tolerance, warning_days}Refresh dari config-sync setiap heartbeat OK
.verified_fpSHA-256 fingerprint saat last verifySet saat heartbeat OK

Logic Status Resolution

HeartbeatService::getStatus() returns:

if (! hasEverVerified())                      → 'unverified'
if (failures < tolerance)                     → 'ok'
if (failures >= tolerance && firstFail age < warning_days) → 'warning'
if (failures >= tolerance && firstFail age >= warning_days) → 'lockout'

Component x-license-banner otomatis:

  • Render banner kalau response.status === 'warning'
  • Render modal kalau response.status === 'lockout'

Customize Poll Interval

Default 5 menit. Untuk ubah, edit license-banner.blade.php baris 110:

// resources/views/components/license-banner.blade.php
var POLL_INTERVAL   = 5 * 60 * 1000;  // 5 menit (default)
// var POLL_INTERVAL = 1 * 60 * 1000;  // 1 menit untuk debugging
// var POLL_INTERVAL = 10 * 60 * 1000; // 10 menit untuk minimasi traffic

var DISMISS_TIMEOUT = 30 * 60 * 1000; // 30 menit setelah dismiss banner muncul lagi
⚠ Trade-off Poll Interval
  • Lebih cepat (1-2 menit): UI lebih responsive saat ada perubahan status. Tapi traffic lebih banyak.
  • Lebih lambat (10-15 menit): Hemat resource. Tapi user harus refresh halaman manual untuk lihat status terbaru.
  • Catatan: poll status TIDAK trigger heartbeat ke Gemilang — ini cuma baca cached state lokal. Aman buat lebih sering.

4. Custom UI Components

Kalau mau tampilan berbeda dari default x-license-banner, panggil endpoint langsung dari kode JavaScript custom:

// Custom polling logic
async function checkLicense() {
    try {
        const response = await fetch('/license/heartbeat-status', {
            headers: { 'X-Requested-With': 'XMLHttpRequest' }
        });
        const result = await response.json();

        if (!result.success) return;

        const { status, days_until_lockout, last_success_at } = result.data;

        // Update your custom UI
        if (status === 'lockout') {
            showLockoutModal();
        } else if (status === 'warning') {
            showWarningBanner(days_until_lockout);
        } else {
            hideAllNotifications();
        }
    } catch (e) {
        console.warn('License check failed', e);
    }
}

// Poll setiap 5 menit
setInterval(checkLicense, 5 * 60 * 1000);
checkLicense(); // initial check

5. Manual Trigger dari Frontend

Tombol "Verify Now" untuk user yang mau force re-check:

// HTML
<button onclick="verifyLicense()">Verify Now</button>

// JS
async function verifyLicense() {
    const csrfToken = document.querySelector('meta[name="csrf-token"]').content;

    const response = await fetch('/license/heartbeat-ping', {
        method: 'POST',
        headers: {
            'X-CSRF-TOKEN': csrfToken,
            'X-Requested-With': 'XMLHttpRequest',
            'Accept': 'application/json'
        },
        credentials: 'same-origin'
    });

    if (response.status === 429) {
        alert('Tunggu 30 detik sebelum mencoba lagi.');
        return;
    }

    const result = await response.json();
    if (result.success) {
        alert('License verified ✓');
    } else {
        alert('Verification failed: ' + result.message);
    }
}
🔒 Rate Limit
Endpoint /license/heartbeat-ping rate-limited 1 request per 30 detik per IP. Untuk mencegah abuse — endpoint ini trigger network call ke Gemilang.

7. Override Heartbeat Behavior per License

Admin Gemilang bisa set tolerance & warning_days berbeda per license (override default global).

  1. Buka /licenses/{hash} di Gemilang admin
  2. Section Heartbeat Policy Override
  3. Set Failure Tolerance + Warning Days → Save

Di sisi backend Gemilang, policy disimpan di master_app_configs dengan app_code = 'license_{license_id}'. Saat config-sync dipanggil, server lookup override ini dulu, baru fallback ke global default.

Di sisi client (ermv3), policy ditarik via config-sync setiap heartbeat sukses, lalu cached di storage/app/licensing/state/.hb_policy (encrypted).

9. Monitoring Heartbeat di Gemilang

Setiap heartbeat dari client dicatat di tabel license_logs_heartbeat. Lihat di:

  • UI Admin: /licenses/{hash} → section "Recent Heartbeats" (20 terakhir)
  • Query langsung untuk monitoring/dashboard:
-- Heartbeat health overview 24 jam terakhir
SELECT
    app_code,
    status,
    COUNT(*) as count
FROM license_logs_heartbeat
WHERE heartbeat_at > NOW() - INTERVAL '24 hours'
GROUP BY app_code, status
ORDER BY app_code, status;

-- Client yang heartbeat-nya delayed
SELECT
    li.hostname,
    li.domain,
    li.last_heartbeat_at,
    NOW() - li.last_heartbeat_at AS delay
FROM license_installations li
WHERE li.status = 'active'
  AND li.last_heartbeat_at < NOW() - INTERVAL '2 hours'
ORDER BY delay DESC;

FAQ

Umum

Apa beda License Key dengan Feature License Key (FLK)?

License Key (LIC-XXXX-XXXX-XXXX-XXXX) adalah kunci utama untuk mengaktifkan aplikasi ERMv3. Tanpa ini aplikasi tidak bisa diakses sama sekali.

Feature License Key (FLK-XXXX-XXXX-XXXX-XXXX) adalah kunci tambahan untuk mengaktifkan fitur premium tertentu di dalam ERMv3. Aplikasi sudah bisa diakses dengan License Key, tapi fitur premium baru aktif setelah FLK dimasukkan.

Berapa lama lisensi berlaku?

Default 365 hari, tapi admin Gemilang bisa atur sesuai kontrak (90, 180, 365, 730 hari, dst). Setelah expired, lisensi bisa di-renew tanpa kehilangan fingerprint atau data.

Apa yang terjadi kalau internet putus?

ERMv3 punya grace period 7 hari. Selama itu aplikasi tetap bisa digunakan. Heartbeat akan retry otomatis. Setelah grace period habis, aplikasi akan locked dan harus reconnect ke Gemilang untuk re-validate.

Apakah lisensi bisa dipindah ke server lain?

Bisa. Caranya:

  1. Admin Gemilang revoke usage server lama (di /licenses/{hash} → Installation Slots → Revoke)
  2. Install ERMv3 di server baru
  3. Aktivasi ulang dengan kunci yang sama

Fingerprint baru akan generate otomatis dari server baru, dan slot lama bisa dipakai server baru.

Untuk Admin

Lupa kunci lisensi yang sudah dikirim ke client. Bagaimana?

Buka /licenses/{hash} → section "Kunci Lisensi Lupa?" → Opsi 1 → klik Tampilkan Kunci. Kunci asli akan muncul di modal (bisa dipulihkan selama APP_KEY Gemilang tidak pernah berubah).

Bagaimana kalau kunci lisensi bocor?

Generate kunci baru via Opsi 2. Kunci lama otomatis tidak valid lagi, semua usage di-revoke. Client harus aktivasi ulang dengan kunci baru.

Bagaimana cara naikkan max_installations?

Saat ini lewat database manual atau via tinker. Edit form belum tersedia di UI:

php artisan tinker
> \App\Models\LicenseCompany::find(1)->update(['max_installations' => 5]);
Client ada 2 server (staging + prod), kunci yang sama bisa?

Bisa, asal max_installations ≥ 2. Setiap server punya fingerprint berbeda, jadi memakan slot terpisah.

Apa yang terjadi kalau saya delete LicenseCompany?

Soft-delete: data tetap di DB tapi tidak aktif lagi. Semua usage hard-delete (slot bebas). Audit log tetap ada di license_logs_audit.

Untuk Developer

Mengapa public key tidak boleh di .env?

Programmer dengan akses repo bisa baca .env (atau .env.example, atau git history). Kalau public key di-leak, attacker bisa pakai untuk verify token palsu. Lebih aman fetch on-demand dari server dan cache encrypted di storage.

Mengapa tidak pakai JWT saja, kenapa PASETO?

JWT punya banyak issue (algorithm confusion, "none" attack, kompleksitas spec). PASETO v4 lebih simpel, sudah hard-code Ed25519, tidak ada algorithm switching, dan secure-by-default.

Bagaimana cara test licensing tanpa Gemilang server live?

Gunakan local Gemilang di port 8001 (atau Docker), arahkan LICENSING_SERVER_URL ermv3 ke http://127.0.0.1:8001. Generate kunci di local Gemilang dan aktivasi ermv3 lokal.

Apakah saya bisa mengubah issuer setelah deploy?

Bisa, tapi semua client harus aktivasi ulang karena PASETO token punya claim iss yang fixed di issuance. Token lama dengan issuer berbeda akan rejected. Setelah ubah issuer di Gemilang config, jalankan revoke all usages, client otomatis akan dapat token baru saat aktivasi.

Apakah heartbeat membebani server?

Tidak signifikan. Default 1 ping per jam per client. Cuma cek fingerprint match di DB + update last_seen_at. Untuk 1000 clients, beban <1 req/detik rata-rata.

Bagaimana memantau heartbeat health di production?

Gemilang otomatis log ke license_logs_heartbeat. Admin bisa lihat di /licenses/{hash} section Recent Heartbeats. Atau query langsung:

SELECT app_code, status, COUNT(*) FROM license_logs_heartbeat
WHERE heartbeat_at > NOW() - INTERVAL '24 hours'
GROUP BY app_code, status;
Apa yang terjadi kalau Gemilang server down?

ERMv3 client masuk ke grace mode (7 hari default). Aplikasi tetap berjalan, tapi:

  • Setelah 3 heartbeat gagal → banner warning di atas aplikasi
  • Setelah 3 hari → modal lockout muncul, tidak bisa ditutup
  • Setelah grace period habis → aplikasi blocked total

Solusi: redundant Gemilang dengan load balancer, atau monitoring SLA Gemilang berbeda dari ERMv3.
